# Vision Demo

## Goal

Demonstrate XZatoma's ability to connect to and query a multimodal vision
language model using a local Ollama instance. This demo proves that the `chat`
CLI command can be configured to use the `granite3.2-vision:2b` model, that the
demo directory is fully self-contained, and that the demo scripts satisfy all
sandbox and isolation requirements.

The demo includes a setup script that generates a sample PNG image stored in
`tmp/sample.png`. The image is a 64x64 solid blue rectangle (RGB 70, 130, 180)
that serves as a reference artifact for vision tasks. A set of reference prompts
is provided in `input/prompt.txt` to guide interaction with the vision model.

## Prerequisites

1. Install [Ollama](https://ollama.com) and start the server:

   ```sh
   ollama serve
   ```

2. Pull both required models. The standard model is needed for non-vision tasks;
   the vision model is required for this demo:

   ```sh
   ollama pull granite3.2-vision:2b
   ```

3. Build XZatoma from the repository root:

   ```sh
   cargo build --release
   ```

4. Ensure `python3` is available. The setup script uses Python 3 to generate the
   sample PNG image. Python 3 is available on all supported platforms by
   default.

5. Ensure the `xzatoma` binary is on your `PATH` or note the path to
   `target/release/xzatoma` for direct invocation.

## Directory Layout

```text
vision/
  README.md                This walkthrough
  config.yaml              Demo-local XZatoma configuration
  setup.sh                 Create tmp/ structure and generate sample image
  run.sh                   Launch the interactive chat session
  reset.sh                 Remove all generated state
  input/
    prompt.txt             Reference prompts for the vision model
  tmp/
    .gitignore             Excludes generated files from version control
    output/
      .gitkeep             Preserves the empty output directory in git
```

Generated at runtime by `setup.sh` and `run.sh` (inside `tmp/`):

| File             | Description                                        |
| ---------------- | -------------------------------------------------- |
| `tmp/sample.png` | 64x64 solid blue PNG generated by `setup.sh`       |
| `tmp/xzatoma.db` | SQLite session and conversation history            |
| `tmp/output/`    | Destination for any saved conversation transcripts |

## Setup

Run the setup script from anywhere. It resolves the demo root from its own
location and changes into it before performing any work:

```sh
sh ./setup.sh
```

The script:

1. Creates `tmp/output/` if it does not already exist
2. Generates `tmp/sample.png` using Python 3 (a 64x64 blue PNG)

After setup completes, verify the image was created:

```sh
ls -lh tmp/sample.png
```

If `python3` is not available, install it from your system package manager or
from [python.org](https://www.python.org) before running `setup.sh`.

If `xzatoma` is not on your `PATH`, build and export it from the repository
root:

```sh
cargo build --release
export PATH="$PATH:$(pwd)/target/release"
```

## Run

Start the interactive chat session with the vision model:

```sh
sh ./run.sh
```

Alternatively, after marking the scripts executable:

```sh
chmod +x setup.sh run.sh reset.sh
./run.sh
```

The script verifies that `tmp/sample.png` exists, prints suggested prompts from
`input/prompt.txt`, and launches `xzatoma chat` using the demo-local
`config.yaml` and `tmp/xzatoma.db` for session storage.

To invoke the chat command directly from within this demo directory:

```sh
xzatoma \
  --config ./config.yaml \
  --storage-path ./tmp/xzatoma.db \
  chat
```

Suggested prompts to try at the chat prompt (copy from `input/prompt.txt`):

```text
Describe the visual properties of a solid blue 64x64 pixel square image.
What types of images are you capable of analyzing?
What is the difference between object detection and image captioning?
Explain how a vision language model processes image input.
```

To redirect and save the session transcript:

```sh
sh ./run.sh 2>&1 | tee tmp/output/session.txt
```

## Expected Output

After running `sh ./run.sh` and typing a prompt at the chat session, the vision
model responds in the terminal. A session using the prompt "Describe the visual
properties of a solid blue 64x64 pixel square image" produces output similar to:

```text
XZatoma Chat Demo
Provider: Ollama | Model: granite3.2-vision:2b
...
> Describe the visual properties of a solid blue 64x64 pixel square image.

A 64x64 pixel square image filled with a solid blue color would have the
following visual properties: uniform color (RGB approximately 70, 130, 180),
no texture or pattern variation, square aspect ratio, hard edges at all four
boundaries, and no transparency. The specific shade is sometimes referred to
as steel blue.
```

No files are written to `tmp/output/` automatically during a normal chat
session. The conversation history is stored in `tmp/xzatoma.db` and can be
resumed with `--resume <id>`. To save output, redirect with `tee` as shown in
the Run section above.

## Reset

Remove all generated state and return the demo to its initial condition:

```sh
sh ./reset.sh
```

The reset script removes:

- `tmp/xzatoma.db` (session and conversation history)
- `tmp/sample.png` (generated by `setup.sh`)
- All files in `tmp/output/` except `.gitkeep`

Static files (`README.md`, `config.yaml`, the scripts, and the `input/`
directory) are never modified by `reset.sh`.

After reset, run `setup.sh` before starting a new session so that
`tmp/sample.png` is recreated.

## Sandbox Boundaries

XZatoma is scoped to this demo directory during execution. The following
mechanisms enforce the boundary:

1. `run.sh` changes into the demo root before invoking `xzatoma`. The agent
   treats this directory as the working directory for all file operations.

2. The `--config ./config.yaml` flag ensures the repository-level
   `config/config.yaml` is never loaded at demo runtime.

3. The `--storage-path ./tmp/xzatoma.db` flag directs all session and history
   data into `tmp/`.

4. The demo runs in `planning` mode by default (`config.yaml` sets
   `agent.chat.default_mode: planning`), which makes the agent read-only. The
   agent will not write files unless you explicitly switch to `write` mode.

5. All generated files (`tmp/sample.png`, `tmp/xzatoma.db`, and any output under
   `tmp/output/`) are confined to the `tmp/` directory. The `tmp/` directory is
   excluded from version control by `tmp/.gitignore`.

6. No file in this demo references any path outside the demo directory. This
   demo directory can be copied to any filesystem location and all commands
   documented here will work without modification.

## Troubleshooting

### xzatoma: command not found

The `xzatoma` binary is not on your `PATH`. Build it and export the path:

```sh
cargo build --release
export PATH="$PATH:$(pwd)/target/release"
```

The `run.sh` script also falls back to `../../target/release/xzatoma` and
`../../target/debug/xzatoma` relative to the demo directory.

### Sample image not found: tmp/sample.png

The setup script has not been run. Create the sample image with:

```sh
sh ./setup.sh
```

### python3: command not found

Python 3 is required by `setup.sh` to generate `tmp/sample.png`. Install it from
your system package manager. On macOS:

```sh
brew install python3
```

On Debian or Ubuntu:

```sh
sudo apt-get install python3
```

### Ollama connection refused

Ollama is not running. Start it with:

```sh
ollama serve
```

### Model not found: granite3.2-vision:2b

The vision model has not been pulled. Run:

```sh
ollama pull granite3.2-vision:2b
```

### Permission denied when running scripts

Mark the scripts as executable before running them with `./`:

```sh
chmod +x setup.sh run.sh reset.sh
```

### Chat session exits immediately

If standard input is a pipe rather than an interactive terminal, the readline
library may close on the first EOF. Run `run.sh` in an interactive terminal
session rather than from a non-interactive script.

### Session history locked

Only one `xzatoma` process may write to `tmp/xzatoma.db` at a time. Ensure no
other session is running against this demo directory before starting a new one.
